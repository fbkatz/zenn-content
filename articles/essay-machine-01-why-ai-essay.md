僕は「マルチエージェント将軍システム」というOSSの上で、日々の開発作業を回している。

yohey-wさんという方が作ったこのシステム（[GitHub](https://github.com/yohey-w/multi-agent-shogun)）が、正直に言って頭がおかしい。戦国時代の軍制をそのままAIエージェントの組織構造に持ち込んでいて、家老がタスク管理AI、その下に足軽と呼ばれる実行AIが8体ぶら下がっている。足軽たちは全員AnthropicのSonnetモデルで、僕が「これやれ」と言えば家老がタスクを分解し、足軽たちが並列で一斉に動き出す。コードを書く、テストを回す、設定ファイルを生成する——そういう作業を毎日この体制でやっている。

で、ある日思ったわけだ。この仕組みでnote.comにコンテンツを量産できないか、と。

毎週1本、エッセイを自動生成してnoteに下書き投稿するパイプラインを作る。僕が軽くテーマだけ決めれば、あとはAIが書いて、APIで下書きに突っ込んでくれる。コンテンツが資産として積み上がっていく。経営者としては実に合理的な発想だ。

ただし条件がひとつある。**AIが書いた感じがしないこと**。

これは見た目の問題じゃない。読んだ人間が「ああ、ChatGPTに書かせたな」と思った瞬間、そのコンテンツの価値はゼロになる。いや、マイナスだ。「この人はAIに書かせた文章を自分の名前で出すのか」という信用毀損が発生する。だから「AIっぽくない」ではダメで、「人間が書いたとしか思えない」レベルが必要になる。

本当にそんなことが可能なのか。正直に言えば、始める前の直感は「たぶん無理」だった。でも無理かどうかは試してみないと分からないし、試す過程で面白いことが分かるかもしれない。少なくとも「なぜ無理なのか」が言語化できれば、それ自体に価値がある。

## なぜエッセイなのか

AIにテキストを生成させること自体は、もう何も珍しくない。コード生成、翻訳、要約、メール文面の作成——実用レベルで使えるタスクは山ほどある。僕自身、毎日足軽たちにコードを書かせているし、その品質には概ね満足している。

でもエッセイは違う。

技術文書やビジネスメールには「正解」がある。情報が正確に伝われば良い、相手が不快にならなければ良い、という明確な評価軸がある。AIはこういう仕事が得意だ。ゴールが定義されていて、そこに最短距離で向かえばいい。

エッセイにはそれがない。

エッセイとは何か、という問いに対して僕なりの答えを言えば、**情報伝達でも論証でもなく、「人間がそこにいる」ことそのものが価値になるジャンル**だと思っている。

東海林さだおのエッセイを読んで、カップ焼きそばの湯切りに関する有益な情報を得たいと思っている人間はいない。筒井康隆のエッセイから論理的に整合した主張を読み取ろうとする人もいない。遠藤周作のエッセイに実用的なアドバイスを期待する読者もいない。

彼らのエッセイが面白いのは、矛盾した人間がそこにいるからだ。東海林さだおは本気でどうでもいいことを本気で考察する。筒井康隆は自分で書いたことを自分でひっくり返す。遠藤周作は深刻な話をしているはずなのにいつの間にか脱線している。その「人間くさい揺らぎ」が読者を惹きつける。

LLMにそれができるか。

LLMの出力は本質的に「もっともらしさの最大化」だ。事前学習で大量のテキストを吸い込み、RLHF（人間のフィードバックによる強化学習）で「人間が好む出力」に最適化されている。つまりLLMは常に「一番無難で、一番平均的で、一番怒られない回答」に収束しようとする。

エッセイに必要な「矛盾した人間の揺らぎ」は、この最適化の真逆にある。矛盾は非効率だし、揺らぎは不安定だ。RLHFが全力で排除しようとするものを、あえて出力させなければならない。

直感的には無理だ。でもまあ、やってみよう。

## 開発体制

今回のプロジェクトでは、足軽2号にプロンプト設計とサンプル生成を任せた。

将軍システムの構造を簡単に説明しておく。

```
僕 ───→ 家老（Claude Code AI）
            │
            ├─ 足軽1号（Sonnet）
            ├─ 足軽2号（Sonnet）← 今回のエッセイ担当
            ├─ 足軽3号（Sonnet）
            │  ...
            └─ 足軽8号（Sonnet）
```

僕が方針を決め、品質を判定する。家老がそれをタスクに分解して足軽に配分する。足軽たちが実行する。全員Anthropicの Claude Sonnetモデルで動いている。

普段はコード実装や設定変更など、明確なゴールがあるタスクを足軽に投げている。今回はちょっと毛色が違う。「面白いエッセイを書け」というのは、足軽にとってはかなり曖昧な指示だ。

ワークフローはこうなった。

1. 僕がプロンプトの方針を決める
2. 足軽2号がプロンプトを設計し、サンプルエッセイを生成する
3. 僕が読んでフィードバックする（「ここがAIっぽい」「ここは面白い」）
4. 足軽2号がプロンプトを修正してリテイクする
5. 2〜4を繰り返す

人間のクリエイティブディレクターとAIライターの関係に近い。僕は一文字も書かないが、品質の最終判断は僕がやる。

## 最初の設計思想——9人のエッセイスト

最初に考えたアプローチは、文体のブレンドだった。

まず9人の日本人エッセイストの文体を足軽に分析させた。

- **東海林さだお**: 断言から入り、細かい観察を積み重ね、最後にボヤく。
- **池波正太郎**: 食の描写が一文一文映像的。日常の所作に美学がある。
- **筒井康隆**: 自分で書いたことに自分でメタツッコミを入れる。
- **星新一**: 短文を連打する。オチがあるようでない。
- **阿川佐和子**: 読者に話しかけるような口調。会話体。
- **池澤春菜**: SFと日常を自在に行き来する。知的好奇心の塊。
- **遠藤周作**: 深刻な話をしているはずがいつの間にか脱線している。
- **ジェーンスー**: 日常を鋭く切り取りながら自分も巻き込む。自虐と社会批評の同居。
- **辻仁成**: 感傷的なディテールの積み重ね。

この9人の文体的特徴をプロンプトに記述し、「これらをブレンドした文体で書け」と指示する。比率も指定できるようにする。東海林さだお40%、筒井康隆30%、星新一30%——みたいに。

我ながら理にかなった設計だと思った。エッセイの「面白さ」を構成要素に分解し、それぞれの要素を持つエッセイストの文体をパラメータとして調合する。エンジニアリング的にはきれいなアプローチだ。

今から思えば、これが最初の過ちだった。

何が間違っていたか。結論から言えば、**面白い文章は要素の調合では作れない**ということだ。東海林さだおの「断言→観察→ボヤき」を抽出しても、それは東海林さだおの面白さの表層でしかない。筒井康隆の「メタ自己ツッコミ」をパターンとして取り出しても、それは筒井康隆がそのツッコミをなぜそのタイミングで入れたかという判断を含んでいない。

文体の特徴量を正確に記述すればするほど、出てくる文章は「特徴量を正確に再現した何か」になる。面白いエッセイにはならない。特徴量の再現はモノマネであって、創作ではない。

でもそれに気づくのは、実際にプロンプトv1を書いて、生成されたサンプルを読んでからだ。プロンプトv1がどれほど几帳面に設計され、そして生成結果がどれほど壊滅的だったか——それは次回に譲る。
